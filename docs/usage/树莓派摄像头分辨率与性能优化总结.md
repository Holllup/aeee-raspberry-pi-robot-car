# 树莓派摄像头分辨率与性能优化总结

## 1. 目标

这次调整的目标是：

- 保留更自然、更宽的摄像头视角
- 不使用裁切
- 提升 OpenCV 实时处理速度
- 在树莓派 3B+ 上找到“画面效果和实时性”的平衡点


## 2. 测试过程中发现的问题

### 2.1 `640x480` 虽然快，但是视角偏窄

最开始测试 `640x480` 时，画面实时性很好，但实际预览发现视角明显偏窄。

原因不是程序额外裁切，而是摄像头本身在这个模式下使用了较小的传感器裁切区域，所以：

- 分辨率虽然低
- 但视角不一定更宽
- 实际上更像“中心裁切后的低分辨率画面”

这对于需要看更宽横向范围的场景不够理想。


### 2.2 `1280x720` 的视角更合适

后续测试 `1280x720` 后，确认这个分辨率的取景效果更符合需求：

- 视角更自然
- 横向信息更多
- 更适合后续做识别和调试

所以最终决定保留 `1280x720` 作为采集分辨率。


## 3. 性能测试结果对比

### 3.1 直接处理 `1280x720`

在以下流程下测试：

- `1280x720` 采集
- 原始 `1280x720` 直接做 OpenCV 处理

处理链路包括：

- 转灰度
- 高斯模糊
- Canny 边缘检测
- 形态学闭运算
- 轮廓提取

测试结果：

- 平均 `13.58 FPS`
- 单帧平均耗时 `73.6 ms`
- 抓帧平均耗时 `22.78 ms`
- 图像处理平均耗时 `50.82 ms`

结论：

- 画面效果可以
- 但实时性偏弱
- 对小车控制这类任务来说会有些慢


### 3.2 `1280x720` 采集，`640x360` 处理

后来采用新的方案：

- 摄像头采集保持 `1280x720`
- 不裁切
- 在程序内部把整张图等比例缩小到 `640x360`
- OpenCV 处理在 `640x360` 上完成

测试结果：

- 平均 `20.55 FPS`
- 单帧平均耗时 `48.64 ms`
- 抓帧平均耗时 `24.4 ms`
- 缩放平均耗时 `6.62 ms`
- 图像处理平均耗时 `17.62 ms`


## 4. 性能提升总结

和“直接处理原始 `1280x720`”相比，这个方案的提升非常明显：

- 平均帧率从 `13.58 FPS` 提升到 `20.55 FPS`
- 提升约 `51%`
- 单帧总耗时从 `73.6 ms` 降到 `48.64 ms`
- 处理耗时从 `50.82 ms` 降到 `17.62 ms`

也就是说：

- 视角保持了 `1280x720` 的效果
- 没有裁切
- 处理速度明显提升
- 更适合树莓派 3B+ 的实时识别场景


## 5. 为什么这个方案更合适

这个方案的优势在于：

- 输入视角由 `1280x720` 决定，所以画面范围不变
- 内部处理分辨率降到 `640x360`，所以计算量大幅下降
- 显示和预览仍然可以基于 `1280x720` 原图进行

这就同时解决了两个矛盾：

- `640x480` 太窄
- `1280x720` 直接处理太慢

因此，“`1280x720` 采集 + `640x360` 处理”是目前最适合的折中方案。


## 6. 当前推荐方案

推荐后续程序采用下面这个模式：

- 摄像头采集：`1280x720`
- 算法处理：`640x360`
- 视角：与 `1280x720` 保持一致
- 不裁切

这个方案比较适合：

- 巡线识别
- 简单颜色识别
- 轮廓/边缘类检测
- 小车实时控制


## 7. 后续建议

如果后面算法继续变重，例如增加：

- 多区域分析
- 更多形态学操作
- 模板匹配
- 神经网络推理

建议继续沿用这个思路：

- 采集分辨率保持不变
- 内部处理分辨率根据性能再调低

例如可以进一步尝试：

- `1280x720` 采集 + `800x450` 处理
- `1280x720` 采集 + `640x360` 处理
- `1280x720` 采集 + `512x288` 处理

这样既能保留视角，又能控制算力消耗。


## 8. 最终结论

本次测试最终得到的最佳平衡方案是：

`1280x720` 采集 + `640x360` 处理

它相比直接处理 `1280x720`：

- 更快
- 更稳
- 视角不变
- 不需要裁切

对于当前这台树莓派 3B+，这是一个非常实用的优化结果。
