# 高分辨率采样，低分辨率处理 1280x720 方案说明

## 1. 方案目标

这个方案的目标是同时满足两件事：

- 保留 `1280x720` 的摄像头视角
- 降低 OpenCV 实时处理的计算量

对应的实现方式是：

- 摄像头采集分辨率保持 `1280x720`
- 程序内部先把整张图等比例缩小到 `640x360`
- 后续识别算法在 `640x360` 上运行

## 2. 这个方案是什么意思

它不是把摄像头直接切换成原生 `640x360` 模式。

它是：

1. 先拿到一张完整的 `1280x720` 画面
2. 再把这张完整画面缩小成 `640x360`
3. 在缩小后的图像上做识别

所以这个方案的核心特点是：

- 视角和 `1280x720` 一样
- 不裁切
- 只是清晰度降低
- 计算量明显下降

## 3. 为什么视角不会变

因为这里做的是 `resize`，不是 `crop`。

两者区别如下：

- `resize`
  只是把整张图缩小，不会丢掉画面的边缘区域，所以视角不变

- `crop`
  只取画面中间或某一块区域，会丢掉边缘内容，所以视角会变窄

只要处理流程是：

```python
full_frame = picam2.capture_array()  # 1280x720
process_frame = cv2.resize(full_frame, (640, 360), interpolation=cv2.INTER_AREA)
```

那么：

- 左右可见范围不变
- 上下可见范围不变
- 只是细节变少

## 4. 为什么不直接用 640x480

前面的测试已经发现，`640x480` 虽然处理更快，但视角偏窄。

这不是程序额外裁切，而是摄像头的低分辨率模式本身可能就是传感器裁切模式，所以：

- 分辨率低
- 但画面范围不一定更大
- 实际看起来反而可能更窄

而 `1280x720` 的取景更符合当前需求，因此更适合作为采集分辨率。

## 5. 这个方案的优势

### 5.1 保留 720p 视角

用户看到的画面范围和 `1280x720` 保持一致。

### 5.2 算法负担更小

`640x360` 的像素数量只有 `1280x720` 的四分之一左右，因此：

- 转灰度更快
- 模糊更快
- 边缘检测更快
- 轮廓提取更快

### 5.3 比直接处理 720p 更适合实时控制

对树莓派 3B+ 这类性能有限的平台，这个方案更容易达到实时处理要求。

## 6. 实测结果

### 6.1 直接处理 `1280x720`

测试方式：

- 采集：`1280x720`
- 处理：`1280x720`
- OpenCV 流程：灰度、模糊、Canny、闭运算、轮廓提取

结果：

- 平均 `13.58 FPS`
- 单帧平均耗时 `73.6 ms`
- 抓帧平均耗时 `22.78 ms`
- 图像处理平均耗时 `50.82 ms`

### 6.2 `1280x720` 采集，`640x360` 处理

测试方式：

- 采集：`1280x720`
- 内部缩放：`640x360`
- OpenCV 流程保持相同

结果：

- 平均 `20.55 FPS`
- 单帧平均耗时 `48.64 ms`
- 抓帧平均耗时 `24.4 ms`
- 缩放平均耗时 `6.62 ms`
- 图像处理平均耗时 `17.62 ms`

## 7. 性能提升

和直接处理 `1280x720` 相比，这个方案的提升很明显：

- 平均帧率从 `13.58 FPS` 提升到 `20.55 FPS`
- 帧率提升约 `51%`
- 单帧平均总耗时从 `73.6 ms` 降到 `48.64 ms`
- 总耗时降低约 `33.9%`
- OpenCV 图像处理耗时从 `50.82 ms` 降到 `17.62 ms`
- 图像处理部分耗时降低约 `65.3%`

这说明：

- 采集仍然保持 `1280x720`，所以视角没有变
- 主要节省下来的时间来自后续图像处理计算量的大幅下降
- 对树莓派 3B+ 这种平台来说，这个优化非常实用

### 7.1 性能对比表

| 方案 | 采集分辨率 | 处理分辨率 | 平均 FPS | 单帧平均耗时 |
|---|---|---:|---:|---:|
| 原方案 | `1280x720` | `1280x720` | `13.58` | `73.6 ms` |
| 优化后 | `1280x720` | `640x360` | `20.55` | `48.64 ms` |

### 7.2 一句话总结

这个优化方案相当于：

`保留 720p 的画面范围，用接近 360p 的计算成本做识别`

## 8. 适合什么场景

这个方案适合：

- 巡线识别
- 颜色识别
- 轮廓检测
- 简单目标检测
- 小车实时控制

尤其适合这种需求：

- 不想损失原本的摄像头视野
- 但又希望算法速度更快

## 9. 典型代码写法

下面是最核心的写法：

```python
from picamera2 import Picamera2
import cv2

picam2 = Picamera2()
config = picam2.create_preview_configuration(
    main={"size": (1280, 720), "format": "BGR888"},
    buffer_count=3,
    queue=False,
)
picam2.configure(config)
picam2.start()

full_frame = picam2.capture_array()  # 1280x720 原始视角
process_frame = cv2.resize(full_frame, (640, 360), interpolation=cv2.INTER_AREA)

# 后续识别逻辑在 process_frame 上执行
gray = cv2.cvtColor(process_frame, cv2.COLOR_BGR2GRAY)
```

## 10. 技术名称建议

这个方案常见的表达方式有：

- 高分辨率采集，低分辨率处理
- 保视角下采样处理
- 采集分辨率与处理分辨率分离

如果用于当前项目，推荐写成：

`高分辨率采集，低分辨率处理（1280x720 采集，640x360 处理）`

## 11. 最终推荐

当前推荐方案是：

- 采集分辨率：`1280x720`
- 处理分辨率：`640x360`
- 处理方式：整张图等比例缩小
- 不裁切

这个方案可以理解为：

`保留 720p 的视角，用 360p 的成本做处理`

对当前这台树莓派 3B+ 来说，这是一个很合适的平衡点。
