## 3. 已确认的 UART 通信参数

当前仓库中，树莓派与底板的车辆控制链路已确认通过 UART 工作。

已确认参数如下：

- 串口设备候选：
  - `/dev/ttyAMA0`
  - `/dev/serial0`
  - `/dev/ttyAMA10`
  - `/dev/ttyS0`
- 当前文档和实测主用串口：`/dev/ttyS0`
- 波特率：`57600`
- Python 串口库：`pyserial`
- 典型超时参数：
  - `timeout=0.1`
  - `write_timeout=0.1`

最小 Python 初始化示例如下：

```python
import serial

ser = serial.Serial(
    "/dev/ttyS0",
    57600,
    timeout=0.1,
    write_timeout=0.1,
)
```

## 4. 已确认的 UART 命令格式

### 4.1 停车命令

停车命令已确认是：

```text
#ha
```

Python 写法：

```python
stop = b"#ha"
```

### 4.2 同时设置方向和速度

当前仓库中实际使用的控制命令格式为：

```text
#ba + 4个方向字符 + 4组16位速度值
```

例如：

```text
#baffff
```

其含义是：

- `#ba`：进入“同时设置方向和速度”的命令模式
- 后面 4 个字符分别对应 4 个电机通道的方向位
- 后面再跟 4 组 16 位速度值，每组 2 字节

### 4.3 方向字符

当前已确认可用的方向字符只有：

- `f`：forward
- `r`：reverse

当前资料不建议使用 `s` 作为方向位来表示停止。单轮停止的推荐方法是：

- 方向位仍写合法值 `f` 或 `r`
- 把该轮速度直接写成 `0`

### 4.4 速度编码方式

每个电机速度使用 16 位编码，按低字节在前、高字节在后的顺序发送。

最小 payload 解释示例如下：

```python
payload = b"#baffff" + bytes([
    40, 0,   # 电机1速度 = 40
    40, 0,   # 电机2速度 = 40
    40, 0,   # 电机3速度 = 40
    40, 0,   # 电机4速度 = 40
])
```

### 4.5 停车的重复发送机制

从当前 Python 控制器实现看，停车不是只发一次：

- `stop()` 会重复发送 `#ha`
- `hold_stop()` 会做一次普通发送

这样做的目的是提高停车命令的可靠性，并清理旧状态。

## 5. 四轮槽位映射

根据当前文档和代码，四个速度槽位的真实顺序已经确认如下：

1. 右前
2. 右后
3. 左前
4. 左后

也就是说，下列速度位对应关系成立：

- `v1`：右前轮
- `v2`：右后轮
- `v3`：左前轮
- `v4`：左后轮

在 Python 控制层，这个映射被抽象成：

```python
RIGHT_MOTOR_INDEXES = (0, 1)
LEFT_MOTOR_INDEXES = (2, 3)
```

这意味着树莓派使用“左右侧速度”来驱动 4 个轮子：

- `right_speed` 会同时写入右前、右后
- `left_speed` 会同时写入左前、左后

这就是当前 `tank drive` 的基础。

## 6. 常用动作语义

### 6.1 前进

四个轮子都前进：

```python
forward = b"#baffff" + bytes([40, 0, 40, 0, 40, 0, 40, 0])
```

### 6.2 后退

四个轮子都后退：

```python
backward = b"#barrrr" + bytes([40, 0, 40, 0, 40, 0, 40, 0])
```

### 6.3 左原地旋转

右侧轮前进，左侧轮后退：

```python
left_spin = b"#baffrr" + bytes([36, 0, 36, 0, 36, 0, 36, 0])
```

### 6.4 右原地旋转

右侧轮后退，左侧轮前进：

```python
right_spin = b"#barrff" + bytes([36, 0, 36, 0, 36, 0, 36, 0])
```

### 6.5 柔和转向

柔和转向不是方向相反，而是左右轮同向但速度不同。

例如柔和左转：

```python
soft_left = b"#baffff" + bytes([40, 0, 40, 0, 20, 0, 20, 0])
```

例如柔和右转：

```python
soft_right = b"#baffff" + bytes([20, 0, 20, 0, 40, 0, 40, 0])
```

## 7. 代码层 UART 发送机制

当前仓库中，`CarMotorController` 的核心工作方式已很明确。

### 7.1 串口选择

控制器会按候选顺序尝试打开串口设备，遇到第一个可成功打开的端口就使用。

### 7.2 发送去重

控制器内部维护 `last_command`。

如果新 payload 与上一次完全相同，普通写入函数会直接跳过，避免重复发送同一命令。

### 7.3 两种停车方式

- `stop()`：强制重复写入 `#ha`
- `hold_stop()`：只发一次 `#ha`

这两个接口通常分别用于：

- 紧急或最终停车
- 控制循环内持续保持停止状态

### 7.4 `set_tank_drive()` 的 payload 构造过程

当前实现流程是：

1. 把 `left_speed` 写入左侧两个电机槽位
2. 把 `right_speed` 写入右侧两个电机槽位
3. 对每个电机根据速度正负号选择方向位
4. 把绝对值速度编码为 16 位低字节在前
5. 组合成完整的 `#ba...` payload 并发送

因此，树莓派侧当前已经具备稳定的 UART 侧写控制能力。

## 8. 与闭环控制直接相关的结论

这是本文档最关键的部分。

### 8.1 当前已确认的事实

当前仓库已经确认：

- 树莓派可以通过 UART 稳定向底板下发 4 路电机控制命令
- 可以完成前进、后退、原地旋转、柔和转向等基础动作
- 当前上位机控制代码的重点是“写命令”，不是“读反馈”

### 8.2 当前未确认的事实

当前仓库没有证实以下任何一点：

- baseboard 是否会通过 UART 返回编码器计数
- baseboard 是否会通过 UART 返回轮速
- baseboard 是否会通过 UART 返回状态帧、里程、故障码或传感器值
- 是否存在 UART 查询命令用于读取编码器或速度
- 是否存在 UART 周期上报码机制

### 8.3 I2C 编码器示例不能直接当作 UART 协议

[车辆控制基础代码.md](/Users/zhwh/Desktop/树莓派/树莓派_GitHub版/docs/hardware/车辆控制基础代码.md) 中确实出现了编码器读取和距离控制相关示例，但这些内容属于：

- `Wire.beginTransmission(...)`
- `Wire.requestFrom(...)`

也就是 `Wire/I2C` 通信，不是当前仓库里已经证实可用的 `UART` 协议。

文档中出现的典型 I2C 示例包括：

- `Wire.write("i0")`
- `Wire.write("i5")`
- `Wire.write("tr")`

因此，当前不能把这些命令直接当作树莓派与 baseboard 的 UART 反馈协议来使用。

### 8.4 当前阶段的正确结论

基于现有仓库，当前只能确认：

- `UART 已证实可以写控制`
- `闭环反馈协议当前未在仓库中证实`

所以，当前仓库足以支撑：

- UART 开环控制
- 基础动作编排
- 强化版原地旋转

但当前仓库还不足以直接支撑：

- 基于 UART 回传轮速的速度闭环
- 基于 UART 回传编码器的角度闭环
- 完整的树莓派上位闭环控制

## 9. 6 线电机闭环开发所需缺口

如果后续目标是“树莓派通过 baseboard 的 UART 通信控制 6 线电机闭环”，当前至少还缺以下外部事实：

- 6 线电机的两根额外线是否真的是编码器反馈线
- 这些反馈线是否已接入当前 baseboard
- baseboard 是否已经采集这些编码器信号
- baseboard 是否会通过 UART 返回编码器计数或轮速
- 若支持回传，UART 协议是否具备：
  - 查询命令
  - 周期上报码
  - 帧头
  - 长度
  - 校验
  - 字节序
  - 通道顺序

如果这些信息缺失，那么即使电机本身是 6 线，也不能自动推导出“树莓派已经能通过 UART 得到闭环反馈”。

## 10. 闭环前置信息清单

在真正实现闭环前，建议先补齐以下信息：

- baseboard 的真实板卡型号
- baseboard 的串口协议文档或厂家示例代码
- 是否存在 UART 读反馈命令
- 串口是否会主动上报数据
- 反馈帧中是否包含 4 个轮子的编码器计数或速度
- 6 线电机额外两根线的引脚定义
- 这两根线在底板上的实际接法

## 11. 后续建议

如果目标是继续推进闭环，推荐按以下顺序进行：

1. 先做 UART 探测脚本或串口抓包，确认底板是否会返回任何可解析数据
2. 如果底板支持 UART 回传，再定义树莓派上的编码器读取层和速度闭环
3. 如果底板不支持 UART 回传，则考虑：
   - 外接反馈传感器
   - 直接读取编码器
   - 在下位机固件中实现闭环
4. 在反馈协议没有确认前，继续把 UART 控制视为“开环控制”

## 12. 当前最终结论

当前仓库已经足够清楚地说明：

- 树莓派到 baseboard 的 UART 写控制链路已经打通
- 电机基础动作控制协议已经明确
- 四轮槽位映射已经明确
- Python 侧控制器实现已经稳定可复用

但同样必须明确：

- 当前仓库没有证实 UART 反馈协议
- 因此暂时不能把“6 线电机闭环”建立在现有 UART 协议已知的前提上

后续闭环开发的第一步，不是继续猜协议，而是先补齐 baseboard 的 UART 反馈能力信息。
