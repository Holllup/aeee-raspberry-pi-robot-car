# 树莓派 3B+ OpenCV 摄像头性能基准测试说明

## 1. 测试目的

这个基准测试用于评估树莓派在连接摄像头后，运行典型 OpenCV 实时视觉处理流程时的性能表现。

它不是单纯测试“摄像头最高帧率”，而是测试：

- 摄像头抓帧速度
- OpenCV 图像预处理速度
- 简单识别任务下的整体实时处理能力

这个测试适合用于估算以下任务的大致性能：

- 巡线识别
- 颜色块检测
- 轮廓识别
- 简单边缘检测
- 传统 OpenCV 实时处理

它不直接代表以下任务的真实性能：

- YOLO、DNN、神经网络推理
- ArUco、SIFT、ORB 等更复杂视觉算法
- 视频编码、存盘、推流
- 多线程异步视觉系统

## 3. 测试依赖

脚本基于以下组件：

- `Python 3`
- `OpenCV (cv2)`
- `numpy`
- `Picamera2`
- 树莓派摄像头驱动栈 `libcamera`

摄像头通过 `Picamera2` 采集图像，不使用旧版 `picamera`。


## 4. 测试流程

脚本会按不同分辨率逐个执行测试，例如：

- `320x240`
- `640x480`
- `960x720`
- `1280x720`
- `1640x1232`
- `1920x1080`

每个分辨率的测试流程如下：

1. 使用 `Picamera2` 按指定分辨率启动摄像头
2. 预热一小段时间，使摄像头输出稳定
3. 在设定时长内连续抓取图像帧
4. 对每一帧执行固定的 OpenCV 处理流程
5. 统计总耗时、单帧耗时、抓帧耗时、处理耗时、FPS 等指标
6. 测试结束后关闭摄像头，切换到下一个分辨率


## 5. 每帧处理逻辑

每一帧图像都会执行下面这套处理流程：

### 5.1 图像采集

通过 `camera.capture_array()` 采集一帧彩色图像，格式为 `BGR`。

### 5.2 转灰度

调用：

```python
cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)
```

作用：

- 降低数据量
- 为后续边缘提取做准备

### 5.3 高斯模糊

调用：

```python
cv2.GaussianBlur(gray, (5, 5), 0)
```

作用：

- 降低噪声
- 减少边缘检测中的毛刺

### 5.4 Canny 边缘检测

调用：

```python
cv2.Canny(blurred, 60, 160)
```

作用：

- 提取图像中的边缘信息
- 模拟常见视觉检测前处理

### 5.5 形态学闭运算

调用：

```python
cv2.morphologyEx(edges, cv2.MORPH_CLOSE, kernel, iterations=1)
```

作用：

- 连通断裂边缘
- 让轮廓更完整

### 5.6 轮廓提取

调用：

```python
cv2.findContours(mask, cv2.RETR_EXTERNAL, cv2.CHAIN_APPROX_SIMPLE)
```

作用：

- 模拟简单目标/区域检测
- 统计面积较大的轮廓数量

这里保留了最基础的“识别”动作，避免测试退化成只做空图像处理。


## 6. 为什么这样设计

这个基准的目标是尽量接近“实际项目中的轻到中等负载视觉任务”，所以没有只测摄像头抓帧，也没有直接上很重的神经网络模型。

这样设计的好处是：

- 比单纯测空帧率更真实
- 比特定项目算法更通用
- 能较好反映树莓派运行基础视觉任务时的瓶颈

从负载强度来看，这个测试大致属于：

- 比“只抓帧”更重
- 比“颜色阈值分割”略重或相近
- 比“深度学习推理”轻很多


## 7. 输出指标说明

每个分辨率会输出一组 JSON 结果，常见字段含义如下：

- `resolution`
  当前测试分辨率

- `frames`
  测试时段内处理的总帧数

- `elapsed_s`
  实际测试持续时间（秒）

- `avg_fps`
  平均每秒处理帧数

- `avg_frame_ms`
  单帧总平均耗时，包含抓帧和处理

- `p95_frame_ms`
  95 分位单帧耗时，反映偶发卡顿情况

- `avg_capture_ms`
  每帧图像采集平均耗时

- `avg_processing_ms`
  每帧 OpenCV 处理平均耗时

- `avg_contours`
  每帧检测到的大轮廓平均数量

- `temperature_c`
  测试时 CPU 温度


## 8. 如何理解测试结果

可以按下面的经验理解：

- `30 FPS` 以上：实时性很好
- `20~30 FPS`：多数实时任务可接受
- `10~20 FPS`：能运行，但会有明显延迟
- `10 FPS` 以下：不适合实时控制类任务

如果你的项目是下面这些场景：

- 巡线
- 小车控制
- 颜色识别
- 简单轮廓检测

建议优先选择能稳定达到 `20 FPS` 以上的最高分辨率。

如果你的项目后续还要增加更重的操作，比如：

- HSV 分割
- 多区域检测
- 模板匹配
- DNN / YOLO

建议在本基准测试推荐结果基础上，再向下降一档分辨率。


## 9. 当前测试的推荐意义

以本次测试脚本的负载模型来说，它主要反映的是：

- 树莓派 3B+ 在传统 OpenCV 任务上的上限
- 哪个分辨率最适合“既实时又保留一定图像信息”

因此，这个基准结果可以作为选分辨率的第一参考，但不是所有项目的最终答案。

如果你后面换成更重的算法，最好再针对具体算法单独补测。


## 10. 可扩展方向

如果后续需要，可以在这个基准基础上扩展出几个版本：

- 轻量版：只测抓帧 + 灰度
- 标准版：当前版本
- 重负载版：增加 HSV、阈值分割、轮廓筛选、目标判定
- 深度学习版：加入 DNN / YOLO 推理耗时测试

这样可以更准确地贴近你的实际项目场景。
